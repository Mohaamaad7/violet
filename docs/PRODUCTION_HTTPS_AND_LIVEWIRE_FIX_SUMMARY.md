# توثيق حل مشاكل بروتوكول HTTPS و Livewire في بيئة الإنتاج (Production)

يوثق هذا الملف تفاصيل المشاكل التي واجهت موقع **Violet** في بيئة الإنتاج عقب إطلاق التحديثات الأخيرة، والخطوات التقنية التي تم اتباعها لحل كافة المشاكل واستعادة عمل الموقع ولوحة التحكم بالكامل.

---

## 📌 ملخص المشاكل التي ظهرت

1. **خطأ Mixed Content في الموقع (Frontend):** المتصفحات قامت بحظر ملفات الـ Assets (CSS و JS) لأنها طُلبت عبر بروتوكول غير آمن `http://` بالرغم من أن الموقع يتم تصفحه عبر `https://`.
2. **فشل التوجيه (404 / Redirect Loops):** حدوث حلقات توجيه لانهائية أو ظهور صفحات 404 عند محاولة فتح لوحة التحكم بعد محاولات إجبار HTTPS.
3. **تعليق لوحة التحكم (Filament Login Infinite Spinner):** بعد عمل الواجهة الأمامية، ظلت لوحة التحكم معطلة مع دوران مؤشر التحميل (الدايرة بتلف) بشكل لانهائي في زر تسجيل الدخول دون إرسال الطلب.

---

## 🛠️ تفاصيل التشخيص والحلول البرمجية

تم تقسيم الحل إلى ثلاث طبقات أساسية لضمان عمل بيئة الإنتاج بسلاسة تحت خوادم **DirectAdmin / LiteSpeed** ومعمارية الـ Reverse Proxy:

### 1️⃣ الطبقة الأولى: إجبار الـ HTTPS وتوثيق البروكسي (Proxy & SSL Offloading)
الخادم يعمل خلف Reverse Proxy (يقوم بتحويل الطلب الداخلي إلى منفذ 80 العادي بعد استقباله مشفراً). هذا يجعل لارافيل يعتقد أن الطلب غير آمن (HTTP). تم حل ذلك برمجياً عبر:

* **تكوين TrustProxies (`bootstrap/app.php`):**
  تم توجيه لارافيل لثقة جميع البروكسيات الواردة لكي يقرأ الترويسات (Headers) مثل `X-Forwarded-Proto` بشكل صحيح:
  ```php
  $middleware->trustProxies(at: '*');
  ```

* **فرض HTTPS جذرياً في بيئة PHP (`public/index.php`):**
  قبل إقلاع الـ Framework بالكامل، قمنا بإجبار البيئة على التعرف على الطلب كـ HTTPS وتعيين المنفذ الافتراضي لضمان توافق مولدات الروابط:
  ```php
  if (isset($_SERVER['HTTP_HOST']) && !in_array($_SERVER['HTTP_HOST'], ['localhost', '127.0.0.1', 'violet.test'])) {
      $_SERVER['HTTPS'] = 'on';
      $_SERVER['SERVER_PORT'] = 443;
  }
  ```

* **فرض المخطط الآمن داخل الخدمة (`app/Providers/AppServiceProvider.php`):**
  تأكيد استخدام بروتوكول HTTPS لجميع عمليات توليد الروابط الداخلية (URL generation):
  ```php
  if (! $this->app->environment('local')) {
      URL::forceScheme('https');
      $this->app['request']->server->set('HTTPS', 'on');
  }
  ```

* **تهيئة خادم الويب (`public/.htaccess`):**
  تعديل كود إعادة التوجيه ليتوافق مع الـ Proxy دون التسبب في Redirect Loop:
  ```apache
  RewriteCond %{HTTPS} off
  RewriteCond %{HTTP:X-Forwarded-Proto} !https
  RewriteRule ^(.*)$ https://%{HTTP_HOST}%{REQUEST_URI} [L,R=301]
  ```

---

### 2️⃣ الطبقة الثانية: معالجة طبقة الكاش (Caching Layer)
بعد تطبيق الحلول البرمجية السابقة، استمر الموقع في طلب الملفات القديمة عبر `http://` بسبب **Spatie ResponseCache** الذي يخزن صفحات الـ HTML كاملة.

* **الحل:** تم إفراغ الكاش المخزن على الخادم باستخدام:
  ```bash
  php artisan responsecache:clear
  ```
  هذه الخطوة جعلت الواجهة الأمامية للموقع تعمل بكفاءة وبدون أي أخطاء Mixed Content.

---

### 3️⃣ الطبقة الثالثة: حل مشكلة تعليق لوحة التحكم (Filament/Livewire Assets)
بعد تفعيل الواجهة الأمامية، بقيت لوحة التحكم Filament عالقة ولا تستجيب لزر تسجيل الدخول مع بقاء مؤشر التحميل ظاهراً تلقائياً فور فتح الصفحة.

* **السبب التقني:**
  كان خيار حقن ملفات Livewire التلقائي معطلاً في ملف الإعدادات (`config/livewire.php`):
  ```php
  'inject_assets' => false,
  ```
  وعندما تم عمل Cache للإعدادات على السيرفر (`config:cache`) تم قفل هذا الإعداد، مما أدى إلى:
  1. عدم تحميل ملفات الـ CSS الخاصة بـ Livewire وبالتالي ظهور مؤشر التحميل (الدايرة) بشكل افتراضي دون إخفائه بـ CSS الخاص بـ `wire:loading`.
  2. فشل تهيئة الجافاسكريبت المرفق مع Livewire/Alpine بشكل كامل، مما منع المتصفح من إرسال طلب تسجيل الدخول (POST request) إلى المسار `/livewire/update`.

* **الحل:**
  تم تعديل الإعداد في `config/livewire.php` إلى:
  ```php
  'inject_assets' => true,
  ```
  ثم تفريغ كاش الإعدادات والـ Views على السيرفر، لتعود لوحة التحكم للعمل بشكل فوري وسليم.

## 📋 الإجراءات الدورية الموصى بها عند الترقية (Deployment Checklist)

لتجنب حدوث أي تعليق أو بقاء روابط قديمة عند رفع تحديثات برمجية أو تعديل في الـ Assets مستقبلاً، يجب دائماً تنفيذ الأوامر التالية على السيرفر بالترتيب:

```bash
# 1. جلب التحديثات البرمجية
git pull origin test

# 2. تنظيف كاش الإعدادات والمسارات والـ Views
php artisan optimize:clear

# 3. إعادة بناء كاش الإعدادات والمسارات
php artisan config:cache
php artisan route:cache

# 4. تفريغ كاش الصفحات المخزن بواسطة Spatie
php artisan responsecache:clear
```
